# 自动教科书形式化：30K Agent 协作完成 500 页研究生数学教材的 Lean 形式化证明

📌 **核心摘要**

Meta FAIR 等机构的研究团队实现了一项多 Agent 系统工程里程碑：使用 30,000 个 Claude 4.5 Opus Agent 在**一周内**完成了一本 500 多页研究生级别代数组合学教科书的 Lean 形式化。最终产出**13 万行 Lean 代码**、**5900 个声明**，覆盖全部 340 个目标定理和定义。总推理成本约**10 万美元**（约 200 美元/页，300 美元/定理），成本已低于人类专家团队薪资估算。该系统采用基于 Git 的 trunk-based 开发模型，通过 Sketcher-Prover-Reviewer 多角色协作、文件系统 Issue Tracker 协调、Pull Request 审核机制，有效解决了大规模 Agent 协作的一致性问题。代码、Lean 代码库及蓝图网站已开源。

---

## 一、研究背景与动机

### 1.1 数学的"审稿危机"

当前数学出版体系面临严峻的验证困境：期刊审稿人在时间和激励结构限制下，往往无法逐条验证论文中的所有证明。正如 Fields 奖得主 Vladimir Voevodsky 所指出的："来自可信作者的技术论证，如果难以检查且与已知正确的论证相似，几乎从不会被详细验证。"这导致数学文献中偶尔会出现错误证明，甚至错误结论。

### 1.2 形式化证明的困境

使用 Lean 等证明助手进行数学形式化是解决这一问题的长期方案——它将信任层缩减到小型验证内核和少量手动审查的定义与定理陈述。然而，形式化面临两大挑战：

| 挑战 | 现状 | 需求 |
|------|------|------|
| **基础内容规模** | Lean mathlib 约 220 万行代码，覆盖本科及部分研究生内容 | 需扩展数个数量级以覆盖研究文献 |
| **形式化成本** | 完全依赖人工，扩展速度近年趋于线性 | 需大幅降低时间、精力和专家门槛 |

### 1.3 自动形式化的三代演进

1. **第一代**：定制化模型形式化孤立数学陈述，在 IMO/Putnam 竞赛题上表现出色，但缺乏仓库级工程能力
2. **第二代**：基于通用模型的单 Agent 编码助手（如 Claude Code），可完成拓扑学教科书部分章节，但无法规模化
3. **第三代**：多 Agent 协作框架——本研究的核心贡献

## 二、核心方法

### 2.1 多 Agent 协作架构

研究团队提出了一套基于软件工程最佳实践的多 Agent Scaffold，核心设计原则：

```
┌─────────────────────────────────────────────────────────────┐
│                    多 Agent 协作系统                          │
├─────────────────────────────────────────────────────────────┤
│  A) 大规模并行：Sub-agents 明确任务分配                       │
│  B) 版本控制：Git + trunk-based 开发，短生命周期特性分支       │
│  C) PR 审核：独立 Reviewer + 清晰代码质量指南                  │
│  D) 合并队列：Staging 分支测试，确保 main 始终可构建           │
│  E) 升级机制：基于文件系统的 Issue Tracker                    │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 Agent 角色设计

| 角色 | 职责 | 输出 |
|------|------|------|
| **Sketcher** | 接收教材章节，形式化所有定义和定理陈述（证明用 `sorry` 占位） | 形式化骨架 |
| **Prover** | 接收待证明定理，填充证明；可升级问题至 Issue Tracker | 完整证明 |
| **Reviewer** | 独立审核 PR：数学审查（陈述准确性）+ 工程审查（代码质量） | 批准/修改/拒绝 |
| **Maintainer** | 处理 Issue Tracker 中的待办事项 | 问题解决 |
| **Triage** | 识别已解决的 Issue 并标记 | Issue 状态更新 |
| **Scan** | 扫描代码库发现全局问题（重复代码、重构机会） | 新 Issue 创建 |
| **Progress** | 跟踪目标定理进度，防止"跑题" | 进度报告 |

### 2.3 协调机制：文件系统 Issue Tracker

研究团队创新性地使用**文件系统**作为 Agent 通信协议：

- 仓库中包含一个文件夹，存放描述待办事项的短文本文件（Issues）
- Agent 被鼓励在 PR 中标记已解决的 Issue
- Reviewer 检查这些判断
- 所有 Agent（包括 Maintainer、Triage、Scan）的 PR 都经过相同的 Review 流程

这种设计避免了定制消息传递系统的复杂性，利用 Agent 天然熟悉的 Git 接口实现协调。

### 2.4 工具集设计

Agent 可访问四类工具：

| 类别 | 工具示例 |
|------|----------|
| **文件工具** | 列出文件、按行号读取/写入/编辑、复制粘贴 |
| **Lean 工具** | Lean REPL 执行、mathlib 文件搜索、声明搜索 |
| **Git 工具** | status/add/log/diff/commit/rebase/reset（仅限自有分支） |
| **Bash 工具** | lake 构建、文本处理（cat/grep/sed 等）、文件系统命令 |
| **Issue 工具** | create-issue、list-issues |

所有工具均设计自动截断机制，防止单次工具使用导致上下文溢出。

## 三、实验结果

### 3.1 案例研究：代数组合学教科书

研究选取 Darij Grinberg 的《Introduction to Algebraic Combinatorics》（2025 年预印本）作为目标：

- **规模**：500+ 页，研究生级别
- **选择理由**：内容大部分不在 mathlib 中、证明完整、外部依赖少、公共领域
- **目标追踪**：340 个定义和定理被指定为形式化目标（排除辅助引理）

### 3.2 核心性能指标

| 指标 | 数值 |
|------|------|
| **运行时间** | 1 周 |
| **Agent 总数** | 30,000 个 Claude 4.5 Opus |
| **代码行数** | 130,000 行 |
| **Lean 声明数** | 5,900 个 |
| **目标覆盖率** | 340/340（100%） |
| **推理成本** | ~$100,000 |
| **单位成本** | $200/页，$300/定理 |

### 3.3 形式化进度分析

系统运行呈现两个明显阶段：

1. **持续扩展阶段**：代码库不断增长（包含不必要的"兔子洞"内容）
2. **清理阶段**：Agent 提示词更新触发代码重构和精简

每个 Commit 代表一个独立 Agent 将其工作合并到主分支。最终所有 340 个目标定理和定义均被完整形式化并证明。

### 3.4 PR 粒度分析

多 Agent 设置确保了 PR 的原子性和细粒度：

- 大多数 PR 约**100 行代码**
- 减少整体代码量的重构 PR 较少见
- PR 频繁在扩展代码库的同时替换旧代码

### 3.5 Token 使用分布

| Agent 类型 | Token 占比 |
|------------|------------|
| Prover | 最高（证明生成密集） |
| Sketcher | 中等（骨架创建） |
| Reviewer | 中等（审核决策） |
| 其他角色 | 较低 |

## 四、技术启示与发展方向

### 4.1 对 AI Agent 开发的启发

**1. 协调问题可通过成熟工程实践解决**

无需复杂的编排逻辑，借用人类软件工程的 Git 工作流（trunk-based + PR + Review）即可实现大规模 Agent 协作。关键洞察：**Agent 已天然熟悉这些工具**。

**2. 文件系统是通用通信协议**

相比定制消息传递系统，基于文件的 Issue Tracker 更简单、可解释、可调试。Agent 可通过读写文件实现状态同步。

**3. 角色分离降低相关性错误**

Proposer/Executor/Checker/Adversary 角色分化可在信息不对称和验证负担下减少相关性错误。

**4. 经济性拐点已现**

$100K 成本已低于人类专家团队薪资估算，且未计入未来效率提升空间。自动形式化正成为经济上可行的选项。

### 4.2 可直接应用的技术点

| 技术点 | 适用场景 | 实施难度 |
|--------|----------|----------|
| Git 分支隔离 | 多 Agent 共享代码库 | 低 |
| 双 Reviewer 机制 | 质量敏感任务 | 中 |
| 文件 Issue Tracker | 跨 Agent 状态同步 | 低 |
| 合并队列 + Staging | 确保主干稳定性 | 中 |
| 工具输出截断 | 防止上下文溢出 | 低 |

### 4.3 局限性与未来方向

**当前局限**：

- 单线程合并队列是瓶颈，PR 需长时间排队
- NFS 带宽限制导致 git worktree 复制和 lake 构建偶发超时
- 需添加延迟和抖动防止"惊群效应"

**改进方向**：

- 自动 Staging 分支 + PR 批处理 + 失败二分法（类似 mathlib 的 bors bot）
- 更高效的分布式构建系统
- 更智能的 Issue 优先级排序

## 五、总结

本研究展示了多 Agent 证明工程系统的有效性：

| 维度 | 成果 |
|------|------|
| **规模** | 130K 行代码，5900 声明，340 目标定理 |
| **速度** | 1 周完成，远超人类验证时间 |
| **成本** | $100K，低于人类专家团队估算 |
| **质量** | 全部目标定理完整证明，通过双 Reviewer 审核 |
| **开源** | 代码、Lean 库、蓝图网站全部公开 |

这是教科书形式化规模和熟练度的新里程碑——从早期本科拓扑学结果和现有库内容重组，迈向研究生教科书的完整独立形式化。随着效率持续提升，自动形式化有望成为数学研究的标准基础设施。
