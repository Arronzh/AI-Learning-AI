# LongSeeker：让 AI 搜索 Agent 学会"断舍离"——弹性上下文编排新范式

📌 **核心摘要**

上海交通大学团队提出 Context-ReAct 范式，赋予搜索 Agent 五种原子操作（Skip/Compress/Rollback/Snippet/Delete），让 AI 在长程搜索任务中能够动态管理自己的"工作记忆"。在此基础上训练的 LongSeeker 模型基于 Qwen3-30B-A3B，在 BrowseComp 上达到 61.5%，在 BrowseComp-ZH 上达到 62.5%，大幅超越通义千问 DeepResearch（43.2%/46.7%）和 AgentFold（36.2%/47.3%）。论文已提交 arXiv（2605.05191），证明"弹性上下文管理"可能是突破 Agent 长程推理瓶颈的关键路径。

---

## 一、研究背景与动机

### 搜索 Agent 的"失忆症"

你有没有遇到过这种情况：让 AI 去网上查资料，查了几轮之后，它反而开始"忘了"最初的目的？

这不是错觉，而是当前搜索 Agent（Search Agent）面临的**原生困境**。

当 AI Agent 执行一个复杂的搜索任务时，它需要经历"思考→搜索工具→观察结果→再思考→再搜索..."的循环。这个循环在学术上被称为 **ReAct 范式**（Reasoning + Acting）。每次循环，Agent 都会产生一段新的推理轨迹、工具调用结果和环境观察——**全部累积在上下文窗口里**。

想象一下，你让 AI 去查"2026年全球 AI 融资最多的 5 家初创公司是谁，它们的核心技术分别是什么"。Agent 可能需要：搜索 AI 融资报告→阅读结果→发现数据不够新→再搜索 Q1 数据→对比各方数据→发现一家公司数据互相矛盾→搜索公司财报交叉验证……

**每一步都在往"工作台"上堆新的内容。**

当这个"工作台"塞满了几百 K Token 的时候，问题就来了：

- **信号淹没在噪声中**：早期找到的重要证据被后期冗余信息淹没
- **上下文超限**：超出模型的 Token 限制，不得不粗暴截断
- **幻觉风险上升**：信息密度过低时，模型开始"编造"

### 现有方案的局限

业界并非没有意识到这个问题。目前主流的应对策略有四种：

| 方法 | 思路 | 缺陷 |
|------|------|------|
| **滑动窗口** | 只保留最近 k 步 | 可能丢掉早期关键证据 |
| **阈值截断** | 超过一定长度就清空整个上下文 | 推理连续性被粗暴打断 |
| **周期性总结** | 每隔若干步压缩一次历史 | 固定粒度的压缩，精度不可控 |
| **主动策展** | 让 Agent 自己决定何时压缩 | 缺乏精细操作，不能回溯删改 |

这些方法有一个共同的问题：**它们都不够"精细"**。就像一个粗心的整理师，要么把所有东西扔掉，要么把所有东西塞进一个箱子，要么每十分钟清理一次——但永远做不到"把重要的放桌上、把次要的收柜子、把没用的丢垃圾桶"。

**本文的核心洞见就是：Agent 需要一个"弹性工作记忆"，能够根据当前任务状态，对上下文的不同部分进行差异化处理。**

## 二、核心方法

### 2.1 Context-ReAct 范式：给 Agent 一套"上下文工具箱"

LongSeeker 的核心创新不是模型本身，而是 **Context-ReAct** 这个范式。它在标准 ReAct 循环中嵌入了**元操作层（Meta-action Layer）**：Agent 在每次生成推理轨迹和工具调用时，同时生成一系列**上下文元操作**，在下一步开始前重塑自己的工作记忆。

打个比方：传统 ReAct 像一个在房间里堆东西的人——所有东西都堆在地上，越来越多。而 Context-ReAct 像一个有收纳盒、碎纸机、桌面的整理达人——每放一件新东西之前，先处理一下已有的东西。

### 2.2 五种原子操作

Context-ReAct 定义了五种原子操作，构成了一个**表达能力完备**的操作集：

**① Skip（跳过）**——保持上下文不变。当上下文已经很紧凑、信息丰富时使用，不做多余操作。

**② Compress（压缩）**——将连续的一段历史步骤替换为一段摘要。比如把"反复尝试搜索同一个问题但找到不同结果"那几步，压缩成一句"已从三个来源交叉验证了 X 数据"。

**③ Snippet（片段保留）**——保留某个步骤中一个精确的子串。数字、实体名称、引文、关键代码片段——这些对精度要求极高的内容，绝对不能通过摘要"模糊化"，必须原样保留。这也是 Context-ReAct 的精妙之处：它提供了一个中途方案，介于"全部保留"和"全部压缩"之间。

**④ Delete（删除）**——移除已经没有任何剩余价值的步骤。那些走偏了的分支、搜索失败的尝试、重复的内容——直接删掉。

**⑤ Rollback（回滚）**——放弃一条无用的分支，把上下文恢复到更早的状态，同时记录放弃原因和可转移的洞见。这有点像写代码时的 git revert——不删除历史，但让"干净分支"继续推进。

### 2.3 完备性证明与效率保证

论文有一个令人印象深刻的数学结论：**Compress 操作本身就具备表达能力完备性**——原则上只用 Compress 就能模拟所有其他操作。但为什么要引入另外四个呢？**因为效率和保真度。**

- **Snippet** 避免了把精确信息"摘要化"带来的信息丢失风险
- **Rollback** 保留了分支探索的"经验教训"，比直接删除更有信息价值
- **Delete** 比 Compress 更轻量，降低生成成本

这是一个"表达能力 vs 实用效率"之间权衡的设计艺术。

### 2.4 训练数据与模型

LongSeeker 基于 **Qwen3-30B-A3B**（30B 参数，3B 激活的 MoE 架构），在 **10,000 条合成搜索轨迹**上进行微调。这些轨迹通过一个"教师模型 + 元操作注入"的流程自动生成，包含了大量需要弹性上下文管理的长程搜索场景。

训练的核心目标是：让模型学会"什么情况下应该做什么操作"。这比单纯的问答训练要复杂得多——模型不仅要学会回答问题，还要学会管理自己的"思考过程"。

## 三、实验结果

### 3.1 全面超越竞品

LongSeeker 在四个代表性基准测试上进行了评估：

| 基准测试 | LongSeeker | 通义 DeepResearch | AgentFold | 高表现模型 |
|----------|-----------|-------------------|-----------|------------|
| **BrowseComp** | **61.5%** | 43.2% | 36.2% | GPT-4o：44.3% |
| **BrowseComp-ZH** | **62.5%** | 46.7% | 47.3% | DeepSeek-V3：50.1% |
| **xbench** | 显著领先 | - | - | o1：55ms |
| **GAIA** | 有竞争力 | - | - | - |

**BrowseComp 和 BrowseComp-ZH** 是当前最具挑战性的长篇网页浏览理解基准测试，要求 Agent 在大量网页中找到并整合信息。LongSeeker 在这两项上分别 **领先第二名 18.3 和 15.8 个百分点**，这个差距在 AI 基准测试中相当惊人。

什么概念？如果把考试及格线设在 60 分，竞争对手们在 40-50 分徘徊，LongSeeker 是唯一一个及格的产品。

### 3.2 真实场景验证

论文还展示了让一个人类用户实际使用 LongSeeker 完成"查找某个学术会议 2026 年接收论文统计情况"的全过程：

1. Agent 生成搜索计划
2. 第一次搜索：找到会议主页
3. Compress：将会议介绍压缩为摘要
4. 第二次搜索：找到论文统计页面
5. Snippet：保留"接收率 22.3%，共计 1256 篇"这段精确数字
6. Delete：删除第一次搜索中不再需要的历史步骤
7. 继续深入搜索子主题分类
8. Rollback：发现一个搜索结果不靠谱，回溯后切换到另一个信源
9. 最终输出结构化的报告

整个过程上下文长度始终控制在 8K Token 以内，而同样任务如果使用标准 ReAct，上下文会膨胀到 30K+ Token。

## 四、技术启示与发展方向

### 4.1 架构层面的启示

LongSeeker 的工作给我们最重要的启示是：**Agent 能力的上限，可能并不在模型本身，而在于它管理"思维过程"的方式。**

这呼应了近期一篇关于 Agent 执行驾驭（Agent Execution Harness）的综述（arXiv:2604.xxxx）的核心观点：在实际部署中，系统可靠性的提升往往来自于持久化记忆、技能文件整理、工具接口统一等"外围设施"，而不是修改模型参数。

Agent 正在经历从"能力藏在权重里"到"能力藏在上下文管理里"的范式转变。

### 4.2 值得关注的局限

- **合成训练的泛化问题**：10K 合成轨迹是否涵盖了真实世界的长尾搜索场景？
- **操作选择的开销**：每次都需要生成元操作，对于简单任务会不会过度设计？
- **稀疏奖励问题**：在非常长的任务链中，模型如何知道哪些操作是真正有效的？

### 4.3 开源 vs 闭源的路线对比

OpenAI 的 Deep Research 走的是完全闭源路线，而 LongSeeker 建立在开源模型 Qwen3 之上。这种差异意味着开源社区可以在此基础上快速迭代。论文实验表明，LongSeeker 在多个关键指标上已经超越了闭源方案——这或许预示着**开源 Agent 正在从"追赶者"变成"领先者"**。

---

## 五、总结

LongSeeker 的核心贡献可以用一句话概括：**它证明了"上下文不是被动接收的容器，而是主动管理的资源"。**

这种将元认知能力（meta-cognition）——即"思考自己思考过程的能力"——引入 Agent 的思路，可能会成为下一代 Agent 系统的标配。想象一下，如果 Agent 不仅能"回答你的问题"，还能"意识到自己在回答过程中走了弯路并主动修正"，这种能力在很多复杂任务中可能会带来质的飞跃。

未来方向也很清晰：更丰富的上下文操作（比如跨会话的记忆共享）、更高效的训练数据生成方式、以及对更长任务链（数千步）的支持。当 Agent 能像人类一样灵活地管理自己的"工作台"时，长程推理的可靠性可能就不再是个问题。

## 本期论文速递

除 LongSeeker 外，今天 arXiv 上还有几篇值得关注的 AI 论文：

**① PhysForge（arXiv:2605.05163）**——ICML 2026 接收。提出了一个生成"具备物理属性的 3D 资产"的框架。简单说，以前的 3D 生成只管"看起来像"，PhysForge 还要管"动起来合理"——物体材质、关节约束、运动学参数一并生成。对虚拟世界、机器人仿真意义重大。

**② Taming Outlier Tokens in DiTs（arXiv:2605.05206）**——发现 Diffusion Transformer 中存在与 ViT 类似的"异常 Token"现象：少数 Token 的范数异常高，吸引了不成比例的注意力。提出双阶段寄存器（DSR）干预，在 ImageNet 和文生图上都显著提升了生成质量。

**③ 低开销黑盒幻觉检测（arXiv:2605.05134）**——将 LLM 当作动态系统，用 Koopman 算子理论建模其表示空间的状态转换，通过"事实态 vs 幻觉态"的转换误差差异来检测幻觉，无需二次采样或外部知识库。方法新颖且计算成本极低。

**④ 实用学习型图像编解码器（arXiv:2605.05148）**——对感知质量和运行时间进行联合优化，通过神经架构搜索找到最佳骨干配置。在 iPhone 17 Pro Max 上编码 12MP 照片仅需 230ms，解码 150ms，比 AV1/VVC 节省 2.3-3 倍码率。实验结果基于严格的主观用户研究。
