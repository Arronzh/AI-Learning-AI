# Multi-LCB：当大模型编程评测从 Python "单语"走向 12 门"多语"时代

📌 **核心摘要**

> LiveCodeBench（LCB）已成为评测大模型代码能力的标杆基准，但它有一个致命局限——只会 Python。现实世界中的软件工程师需要同时驾驭多种编程语言，一个 Python 高手可能对 C++ 或 Java 一窍不通。来自 GigaCode 与 Yandex 数据分析学院的研究团队提出了 **Multi-LCB**，将 LCB 评测从纯 Python 拓展至 12 种编程语言（Python、C++、Java、Go、Rust、JavaScript、TypeScript、Kotlin、Scala、Ruby、C#、Swift），在保持 LCB 抗污染体系的同时，揭示了 LLM 在多语言编程能力上的严重不平衡。本文已被 **ICLR 2026** 接收。

## 一、研究背景与动机

### 从 HumanEval 到 LiveCodeBench：代码评测的演进

代码生成是大语言模型（LLM）最受关注的应用之一。从 OpenAI Codex 到 DeepSeek Coder，再到 Qwen-Coder 系列，模型不断刷新编程成绩。伴随模型演进的还有评测基准的迭代：

- **HumanEval / MBPP**：开创性的 Python 函数级评测，但题目量小、静态且容易过拟合；
- **APPS / CodeContests**：增加了竞赛级难度，但仍局限于 Python；
- **LiveCodeBench（LCB）**：由 Jain 等人于 2024 年提出，通过持续从 LeetCode、AtCoder、Codeforces 等竞赛平台抓取新题，按发布日期过滤来避免数据污染，迅速成为业界标准。

### 一个被忽视的问题

然而 LCB 仅评测 Python，这映射出整个领域的结构性偏见：

> **Python 表现好 ≠ 多语言能力强。**

在真实的软件工程中，C++ 用于系统编程，Java 用于企业级应用，JavaScript/TypeScript 用于 Web 开发，Swift 用于 iOS 生态。如果一个 LLM 只在 Python 上表现优异，而在其他语言上大幅滑坡，其实际工程价值就要打上问号。

"当前的评测完全无法回答一个关键问题：LLM 的代码能力是真正理解了编程范式，还是仅仅是 Python 的过拟合？"论文第一作者 Maria Ivanova 在论文中这样写道。

### 12 种语言覆盖

Multi-LCB 选择了覆盖主流应用场景的 12 种语言：

> Python, C++, Java, Go, Rust, JavaScript, TypeScript, Kotlin, Scala, Ruby, C#, Swift

这些语言既有动态类型也有静态类型，既有编译型也有解释型，涵盖函数式、面向对象和系统级编程范式，构成了一个全面且具有挑战性的评测矩阵。

## 二、核心方法

### 从函数签名到 STDIN/STDOUT：解决跨语言评测的翻译难题

多语言评测最大的技术挑战是：**同一个算法问题在不同语言中有着截然不同的语法和测试方式。**

以前的方案（如 MBXP、MultiPL-E、HumanEval-XL）需要为每种语言单独编写函数签名和单元测试。例如，一个简单的二项式系数测试：

```
# Python
assert binomial_coeff(5, 2) == 10

# Java
assertEquals(10, Main.binomialCoeff(5, 2));
```

这种逐语言重写的方案不仅工程量大，还引入了语言间不对等——同一次测试在 Python 中是 3 行代码，在 Java 可能变成 20 行。

**Multi-LCB 的解决方案：统一使用标准输入/标准输出（STDIN/STDOUT）格式。**

所有评测题目不再依赖函数签名测试，而是统一描述为：

```
输入：
5 2

输出：
10
```

模型不论使用哪种语言，只要能从 stdin 读取输入，向 stdout 输出结果即可。这一巧妙设计使得：

1. **12 种语言共享同一套评测集**，无需为每种语言重写测试；
2. **自动继承 LCB 的持续更新**，新题出现时自动可用；
3. 避免了函数签名翻译带来的偏差和额外负担。

### 评测流程

Multi-LCB 保持了 LCB 的核心设计哲学：

1. **时序过滤**：只保留题目发布日期早于模型训练数据截止日期的题目，防止数据污染；
2. **持续更新**：自动从 LeetCode / AtCoder / Codeforces 拉取最新题目；
3. **pass@k 指标**：每个题目生成 k 次，取任意一次通过的概率。

### 被评测模型

研究团队评测了 24 个大模型，涵盖指令微调（Instruction）和推理增强（Reasoning）两大路线，包括：

- **开源模型**：DeepSeek Coder V2、CodeLlama、Qwen-Coder 系列、StarCoder2 等
- **闭源模型**：GPT-4o、Claude 3.5 Sonnet、Gemini 系列等

## 三、实验结果

### 关键发现 1：Python 不是多语言能力的可靠代理

最直观的结论：**Python 排名与整体多语言排名存在显著差异。**

一些在 Python 上表现顶尖的模型，在其他语言中的位置大幅下滑。例如，某模型在 Python pass@1 上位列前三，但在 Java 和 C++ 上跌出了前八。这意味着依赖 Python-only 评测会高估模型的通用编程能力。

### 关键发现 2：Python 过拟合普遍存在

研究团队发现一个系统性模式：**Python 上的出色表现常常伴随着其他语言上的性能断层。** 

在 24 个被评测模型中，超过半数出现了不同程度的 Python 过拟合——即 Python 成绩远高于多语言平均成绩，差距在 15% 以上。

### 关键发现 3：语言特定污染问题

一个更有趣的发现是：**数据污染在不同语言中的分布极不均匀。**

某些模型在 C++ 和 Java 上的表现异常好，远超其在其他语言上的水准，这暗示预训练语料中 C++ 和 Java 的代码占比更高。相比之下，Rust、Swift、Scala 等相对小众语言的表现普遍偏低，模型在这些语言上缺乏足够的训练样本。

### 关键发现 4：显著的多语言能力差距

考察每种语言的平均通过率，存在一个鲜明的分层：

| 梯队 | 语言 | 平均 pass@1 |
|------|------|-------------|
| 第一梯队（强） | Python, C++, Java | 60%+ |
| 第二梯队（中） | Go, JavaScript, TypeScript, C# | 40-55% |
| 第三梯队（弱） | Rust, Kotlin, Ruby, Swift, Scala | 25-40% |

这一结果揭示了一个残酷的现实：**当前 LLM 的"多语言编程能力"严重向语料丰富的工业语言倾斜。**

### 推理模型 vs 指令模型的对比

评测还发现，推理增强模型（Reasoning）相比指令微调模型（Instruction），在多语言场景下的优势更加显著。推理模型通过链式思维和逐步验证，能够更好地适应不熟悉语言的语法和标准库差异。

## 四、技术启示与发展方向

### 对模型开发者的启示

1. **多语言评测不应缺席**：仅仅使用 Python 评测来推断模型的代码能力是不够的。开发者在模型选型时，应根据自身应用场景关注目标语言的专项成绩。

2. **数据配比需要重新审视**：Python 过拟合的根源在于训练数据中 Python 代码占比过高。重新平衡不同语言的训练数据配比，可能是在不牺牲 Python 性能的前提下提升多语言能力的最直接路径。

3. **推理能力有助于跨语言泛化**：推理增强模型在多语言场景下的优势提示我们，让模型学会通用的算法思维（而不只是特定语言的 API 记忆），是提升泛化能力的关键。

### 对评测社区的贡献

Multi-LCB 的 STDIN/STDOUT 范式被证明是一种轻量级且可自动扩展的多语言评测方案。它完全兼容 LCB 的持续更新机制，意味着任何新增的 LCB 题目会自动加入到所有 12 种语言的评测中。这种设计将极大降低多语言评测的维护成本。

### 局限与未来方向

研究团队也坦诚指出了当前工作的局限：

- 12 种语言虽然覆盖面广，但仍有遗漏（如 Lua、R、Julia、Fortran 等）；
- 每题只使用 STDIN/STDOUT 格式评测，未能覆盖语言特定的 API 调用、多文件工程和模块化编程；
- 目前的评测集以竞赛题为主，与实际工业级项目仍有差距。

未来方向包括：引入多文件项目构建、评测代码审查和调试能力、以及支持更多小众语言。

## 五、总结

Multi-LCB 向我们揭示了一个重要但长期被忽视的事实：**LLM 的多语言编程能力远未成熟。** Python 是第一语言，但不应该是唯一的评测语言。

对开发者而言，这意味着在选择"编程助手"时需要更审慎：它在你的目标语言上表现如何？对学术研究者而言，这意味着多语言代码生成从评测方法到数据配比再到模型架构，仍是一片亟待深耕的领域。

**Multi-LCB 不只是一个基准，更是一面镜子——照出了当前大模型代码能力评测的盲区。**
